# Мультифайловая Фабрика Агентов v14.0

## Что это?

Это оркестратор на Python, который автоматизирует полный цикл разработки программного обеспечения (SDLC) с помощью больших языковых моделей (LLM). Вместо одного промпта система разбивает задачу на роли:

- **Business Analyst** – формирует бизнес-требования и user stories.
- **System Analyst** – создаёт техническую спецификацию (модели данных, компоненты, бизнес-правила).
- **Architect** – проектирует архитектуру, перечисляет файлы и зависимости.
- **Arch Validator / DevOps / System Analyst (вторично)** – многоуровневая валидация архитектуры.
- **Developer** – пишет код для каждого файла с учётом глобального контекста.
- **Self-Reflect** – разработчик проверяет себя перед внешним ревью.
- **Reviewer** – проводит code review (SOLID, безопасность, импорты).
- **E2E Architect / E2E QA** – параллельное статическое ревью всего проекта.
- **QA Runtime** – анализирует рантайм-ошибки и предлагает фиксы.
- **DevOps Runtime** – исправляет окружение, зависимости, Dockerfile.
- **Test Generator** – генерирует unit-тесты.
- **Documenter** – пишет README.md.
- **Supervisor** – оркестратор, решает, какую фазу запустить следующей.

Все агенты общаются через структурированный JSON, а код исполняется в изолированном Docker-контейнере, что гарантирует воспроизводимость и безопасность.

## Зачем это нужно?

- **Эксперименты** – исследование возможностей LLM в автоматизации разработки.
- **Быстрое прототипирование** – из текстового описания получаем рабочий код с тестами и документацией.


## Как это работает?

1. **Инициализация**  
   - Пользователь вводит имя проекта, язык (Python/TypeScript/Rust) и описание задачи.
   - Агенты последовательно генерируют требования, спецификацию и архитектуру.
   - Архитектура проходит трёхуровневую валидацию (системный аналитик, валидатор, DevOps).
   - Создаются файлы зависимостей (`requirements.txt`, `package.json`, `Cargo.toml`), структура папок и `ARCHITECTURE.md`.

2. **Основной цикл (управляется Supervisor)**  
   Supervisor анализирует состояние (сколько файлов одобрено, пройдены ли фазы, счётчики ошибок) и выбирает следующую фазу:
   - **develop** – разработка/доработка файлов (с self-reflect и reviewer).
   - **e2e_review** – параллельное E2E-ревью (архитектор + QA).
   - **integration_test** – запуск в Docker, обработка рантайм-ошибок с помощью QA Runtime и DevOps.
   - **unit_tests** – генерация и прогон тестов, проверка покрытия.
   - **document** – генерация README.
   - **revise_spec** – пересмотр спецификации при системных противоречиях.
   - **success** – фиксация результата, генерация SUMMARY.md, предложение доработки.

3. **Обработка ошибок и обучение**  
   - Ведётся статистика успехов/неудач для каждой модели (`model_stats.json`).
   - Результаты LLM кэшируются (`llm_cache.json`), чтобы ускорить повторные запуски.
   - При падении фазы более 3 раз подряд Supervisor инициирует пересмотр спецификации.
   - Система сохраняет состояние после каждой итерации, позволяя продолжить с места остановки (даже после Ctrl+C).

4. **Интеграция с Docker**  
   - Каждый запуск (приложение, тесты) происходит в Docker для чистоты окружения.
   - Автоматически подбирается базовый образ в зависимости от языка.
   - При ошибках сборки или выполнения DevOps-агент может предложить патчи для Dockerfile или альтернативные пакеты.

## Ключевые особенности

- **Мультиязычность** – встроенная поддержка Python, TypeScript, Rust. Легко расширяется добавлением новых образов и команд.
- **Саморефлексия** – перед отправкой на внешнее ревью разработчик проверяет свой код через агента `self_reflect` (temperature=0), что снижает количество очевидных ошибок.
- **Параллельное E2E-ревью** – архитектор и QA проверяют проект одновременно (через `ThreadPoolExecutor`), ускоряя обратную связь.
- **Валидация архитектуры** – три независимых агента оценивают предложенную архитектуру, что повышает качество начального проекта.
- **Умный Supervisor** – принимает решение на основе не только счётчиков, но и флагов (`e2e_passed`, `integration_passed` и т.д.), предотвращая преждевременный переход к финалу.
- **Устойчивость к сбоям LLM** – механизм повторных попыток с экспоненциальной задержкой, fallback на извлечение JSON из текста, поддержка `json_repair`.
- **Git-интеграция** – автоматические коммиты после значимых изменений (инициализация, успешная сборка).

## Архитектура кода

- **`PipelineContext`** – контейнер для глобального состояния, обеспечивающий сохранение при прерывании.
- **`ask_agent`** – унифицированный вызов LLM с кэшированием, повторными попытками и логгированием.
- **Фазы** – отдельные функции (`phase_develop`, `phase_e2e_review`, …), каждая получает `logger`, `state`, `cache` и модифицирует состояние.
- **Утилиты** – работа с Docker, файлами, извлечение публичного API, построение порядка файлов по зависимостям, поиск ошибочного файла по traceback.
- **Статистика** – класс `ModelStats` записывает успешность каждой модели по ролям.

## Как использовать

1. Установите зависимости: `pip install openai json_repair` (последнее опционально).
2. Убедитесь, что работает локальный LLM-сервер (например, Ollama) с нужными моделями.
3. Запустите скрипт: `python factory.py`.
4. Ответьте на вопросы:
   - Подтвердите запуск.
   - Выберите режим ротации моделей (случайный или циклический).
   - Введите имя проекта (латиница).
   - Выберите язык.
   - Опишите задачу.
5. Наблюдайте за процессом в консоли. Логи пишутся в `agent_interactions.log`.
6. При успехе получите папку проекта с готовым кодом, тестами, `README.md`, `SUMMARY.md` и `ARCHITECTURE.md`.
7. Можно продолжить с прерванного места или доработать вручную (опция после success).

Выход:  
- структура проекта с `main.py`, `calculator.py`, `tests/`, `requirements.txt`.
- код с Flask-приложением, логикой калькулятора, тестами pytest.
- `README.md` с инструкцией по запуску.
- Dockerfile для контейнеризации.
- Всё это после нескольких итераций, проверок и исправлений.
